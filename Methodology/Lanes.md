# Полосы (Lanes)

## Trivial

Использовать только для:
- Исправления опечаток
- Переименований
- Мелких правок конфигурации
- Локальных однофайловых изменений без архитектурного воздействия

Поток: `edit → review`

НЕ используйте Trivial, если применимо хотя бы одно из следующего:
- Изменяется публичное поведение
- Затронуто несколько модулей
- Существует риск регрессии
- Есть последствия для безопасности/приватности/хранилища

## Professional

Полоса по умолчанию для большинства задач.

Использовать для:
- Средних и крупных фич
- Многофайловых рефакторингов
- Новых возможностей
- Координации UI + управления состоянием + домена
- Изменений интеграции

Поток:
```
idea → prd → research → vision → plan → implement → review → qa
```

## Critical

Обязательна для работы, связанной с:
- Аутентификацией и авторизацией
- Криптографическими операциями
- Обработкой секретов/ключевого материала
- Безопасным хранилищем и шифрованием
- Миграциями данных
- Подписанием и построением транзакций
- Изменениями API-контрактов
- Обработкой финансовых данных

Поток:
```
idea → prd → research → vision → plan → implement → review → security review → qa
```

### Правила полосы Critical
- Пакеты меньше, чем в Professional
- Пробелы в проектировании должны быть закрыты до реализации
- Предположения, чувствительные к безопасности, должны быть явно указаны в плане
- Ревьюер должен подготовить фазу для security review
- Нельзя переходить к QA, если security review возвращает блокировку

## Проверки качества по полосам

Все полосы запускают `/aidd-run-checks`. Конкретный набор проверок зависит от Tech Adaptor:

| Проверка | Назначение | Примечания |
|---|---|---|
| Format | Форматирование кода | Только изменённые файлы, приоритет MCP |
| Analyze | Статический анализ | Нулевая толерантность (фатальные warnings + infos) |
| Lint | Дополнительные правила линтинга | Зависит от стека (DCM, ESLint и др.) |
| Test | Юнит/интеграционные тесты | Все тесты должны проходить |

Critical дополнительно требует прохождения гейта `security-reviewer`.

## Практическое правило

Если вы не уверены между Professional и Critical, выбирайте Critical.
Накладные расходы — один дополнительный проход ревью; стоимость пропущенной проблемы безопасности значительно выше.

## Professional Short

Используется только когда выполнены ВСЕ следующие условия:

- Изменение только UI (визуальная верстка, дерево виджетов, стили, правки текстов)
- Затронуто не более 2 модулей
- Нет миграций базы данных или хранилища
- Не затрагиваются поверхности: security, storage, domain-API или auth
- Нет изменений публичных контрактов (нет добавления, удаления или breaking переименований API)

Набор артефактов: PRD содержит только Goal + Scope + AC; Research опционально; Vision опционально; security-reviewer отсутствует; spec-critic сохраняется.

Количество фаз: 1–3 (жёсткий лимит; превышение 3 фаз вызывает реклассификацию в Professional).

Правило реклассификации: если в любом батче обнаружена поверхность security, storage, domain-API или auth, которая не была предусмотрена при создании тикета, тикет немедленно реклассифицируется в Professional или Critical. Реклассификация фиксируется в tasklist и phase brief перед продолжением работы.

## Бюджет фаз (Phase budget)

Максимальное количество фаз по полосам:

| Полоса | Лимит фаз | Примечание |
|--------|-----------|------------|
| Critical | Без ограничений | Определяется риском и границами безопасности |
| Professional | Мягко 3–5 | При превышении 5 — рассмотри разбивку на второй тикет |
| Professional Short | 1–3 | Жёсткий лимит; превышение 3 вызывает реклассификацию в Professional |
| Trivial | 0 фаз | Один коммит; без структуры фаз |
