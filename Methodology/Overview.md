# Обзор рабочего процесса

## Модель

`Claude-Native Enterprise AIDD v3` — это полноценный инженерный рабочий процесс для Claude Code:

- Декомпозиция фичи на фазы
- Планирование фазы с выделенными ролями
- Пакетная реализация с контрольными точками утверждения
- Независимое ревью и QA
- Отдельное ревью безопасности для рискованной работы
- Валидатор как страж процесса
- MCP для детерминированных операций
- Claude hooks как основные ограждения
- Выборочные субагенты и командный режим для масштабирования
- Слой оптимизации с учётом токенов

## Полосы (Lanes)

| Lane | Когда | Поток |
|---|---|---|
| `Trivial` | опечатка, переименование, мелкая правка конфига | edit → review |
| `Professional` | средние/крупные фичи, рефакторинги, новые возможности | idea → prd → research → vision → plan → implement → review → qa |
| `Critical` | авторизация, криптография, секреты, хранилища, миграции, API-контракты | поток Professional + security review |

Полоса по умолчанию: `Professional`.
Обязательная `Critical` для: аутентификации, криптографических материалов, обработки секретов/ключей, безопасного хранилища, миграций данных, подписей, изменений API-контрактов.

Подробнее см. [[Lanes]].

## Полный жизненный цикл

```
IDEA_READY
→ PRD_READY
→ RESEARCH_DONE
→ VISION_APPROVED
→ PLAN_APPROVED
→ TASKLIST_READY
→ IMPLEMENT_STEP_OK
→ REVIEW_OK
→ SECURITY_REVIEW_OK (только Critical)
→ QA_PASS
→ RELEASE_READY
→ DOCS_UPDATED
```

Каждый гейт является блокирующим. Следующая роль начинает работу только после прохождения текущего гейта.
Подробнее см. [[Roles And Gates]].

## Оптимизация с учётом токенов

| Техника | Где применяется | Эффект |
|---|---|---|
| `async: true` | Уведомительные хуки (6 из 12) | Неблокирующие, экономия токенов рассуждения |
| `statusMessage` | Тяжёлые хуки (4 из 12) | Спиннер в UI, 0 токенов в контексте |
| `context: fork` | validate, run-checks | Вывод остаётся вне основного контекста |
| `effort: low/medium` | Механические скиллы | ~30-50% экономии на рассуждении |
| `paths` | Доменные скиллы | Загрузка только при работе с релевантными файлами |
| MCP `paths` | формат run-checks | Форматирование только изменённых файлов |
| Компактные сообщения | Все systemMessages | 1 строка, <20 токенов |

Чистая экономия: ~3200 tok/сессия.
Подробнее см. [[Token Optimization]].

## Автоматические ограждения

| Ограждение | Как работает |
|---|---|
| PostCompact | Автовосстановление тикета/фазы/полосы/цели после компактификации контекста |
| PreToolUse gate | Запрещает правки в исходных директориях без активного тикета |
| PostToolUse format | Автоформатирование исходных файлов после каждой правки (через Tech Adaptor) |
| FileChanged validator | Ревалидация слоя процесса при изменении файлов рабочего процесса |

## Почему не Lean-First

Для сложных продуктов ошибки в архитектуре, безопасности, хранилище, доменных контрактах и интеграционном поведении слишком дороги. Процесс создан не ради скорости. Скорость достигается за счёт:

- Хороших шаблонов
- Чётких ролей
- Claude-native среды выполнения
- Валидатора
- Пакетного выполнения
- Оптимизации токенов
- Отсутствия дублирования смысла между документами
