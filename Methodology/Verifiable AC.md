# Verifiable AC

Workflow Minor: 3.3
Owner: methodology

Каждый критерий приёмки в PRD поставляется как `test:` / `command:` / `manual:` shell-проверяемый предикат. Ниже зафиксированы устойчивые формы AC, выкристаллизовавшиеся из практики spec-critic.

## Verifiable AC — анти-паттерны

Доминирующий анти-паттерн — литеральный grep по токену: AC, который проверяет наличие конкретного слова, заголовка или имени секции, ломается на любом безобидном переименовании или переформулировке, даже если артефакт всё ещё удовлетворяет исходному намерению. На цикле критики `BW-XXXX` это случилось напрямую: исходные `AC-A`, `AC-B`, `AC-C`, `AC-D` все кодировали имена секций дословно и потребовали Route 2 (ревизия PRD), прежде чем гейт смог закрыться.

Заменяйте литеральный grep одним из трёх устойчивых вариантов:

- **count-based** — проверяйте структурный счётчик, а не конкретный токен. Пример: `grep -c "^## " <file>` ≥ N. Переживает переименование секций.
- **file-existence** — проверяйте, что артефакт лежит по контрактному пути. Пример: `test -f docs/<TICKET>/discovery/<TICKET>-phase-<N>-discovery.md`. Переживает правки содержимого.
- **семантическое сравнение** — проверяйте равенство двух множеств (или включение одного в другое) через `comm`. Пример: `comm -23 <(printf '%s\n' <expected sections> | sort) <(grep "^## " <file> | sed 's/^## //' | sort)` возвращает пусто. При провале сам называет недостающий элемент.

## Anti-leak AC — каноническая форма

Anti-leak AC для любого vault- или кросс-репозиторного diff sweep ОБЯЗАН быть выражен двумя ногами под `set -e`:

```sh
set -e
# keyword leg — запрещённые доменные токены
git diff <SHA> | grep -iE "^\+.*(<token1>|<token2>|<token3>)" | wc -l   # ДОЛЖНО быть 0
# ticket leg — только allowlisted placeholder-ы тикетов
git diff <SHA> | grep -v -E "<allowlist>" | grep -E "\bBW-[0-9A-Z]+\b" | wc -l   # ДОЛЖНО быть 0
```

И keyword leg, и ticket leg должны независимо возвращать нулевой счётчик. Single-pipe compound форма (`... | grep keywords | grep tickets`) запрещена: она позволяет утечке не-тикетного keyword проскочить, потому что финальный ticket-grep отбрасывает строку. Форма two-leg — единственная допустимая для anti-leak AC.

## Гигиена exclusion-паттернов

Когда AC использует `grep -v` для отсечения исторических или намеренно сохраняемых ссылок, exclusion-паттерны ОБЯЗАНЫ якорить каждое имя файла к его охватывающему сегменту директории. Используйте `/phase/phase-3.md`, а НЕ голое `/phase-3.md` — неякорная форма заматчит также `phase-30.md`, `phase-3.md.bak` или любой случайный `phase-3.md`, положенный позже под непредусмотренный подкаталог. Это и есть анти-паттерн широких exclusion-ов.

> Exclusion-список исчерпывающий; если устаревшая ссылка всплывает под не-исключённой поддиректорией — чините артефакт, а не расширяйте exclusion. Дрейф оператора в сторону более широких exclusion-ов прячет регрессии, которые AC и был построен ловить.

## Маршруты ремедиации AC

Когда `command:` AC падает на первом sweep'е, спросите: дрейфнула ли реализация, или AC предсказал не ту форму? Это и есть выбор маршрут ремедиации.

- **Route 1** (правка содержимого под AC) — если AC описывает задуманную форму, а артефакт её не достиг.
- **Route 2** (ревизия AC) — если артефакт корректен, а AC был over-specified.

Route 1 — по умолчанию; Route 2 требует одной строки обоснования в следе критики, чтобы история ревизий оставалась аудируемой и будущие авторы видели, какие AC ужесточались, а какие ослаблялись.

## Каденс spec-critic

Два прохода spec-critic — ожидаемая каденса для PRD среднего размера, а не сигнал тревоги. Первый проход обычно фиксирует литеральные анти-паттерны и пропущенные anti-leak ноги; второй — закрывает остаточные правки формулировок. Третий проход — повод проверить, не слишком ли широк был исходный PRD; рассмотрите декомпозицию фазы.
