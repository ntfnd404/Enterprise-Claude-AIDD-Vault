# Контракты артефактов

## Основной принцип

**Один документ = одна ответственность.** Ни один артефакт не дублирует область другого.

## Два слоя

### Постоянный слой (`docs/project/`)

Остаётся в основной ветке. Хранит:
- Conventions (архитектурные правила, запреты)
- Workflow (определение процесса)
- Code style guide (форматирование, именование)
- Guidelines (паттерны, специфичные для фреймворка)
- ADR (записи архитектурных решений)
- Templates (канонические формы артефактов)

НЕ хранит: локальные для фазы рассуждения, одноразовые детали реализации, QA-отчёты для конкретных фич.

### Слой фичи (`docs/<TICKET>/`)

Локальный для ветки, очищается перед мержем. Хранит все артефакты фаз одной фичи.

## Карта артефактов

| Артефакт | Владеет | НЕ должен владеть |
|---|---|---|
| **Idea** | Проблема, скоуп, намерение, зависимости, приёмка, полоса | Дизайн реализации |
| **PRD** | Deliverables, сценарии, метрики, ограничения | Пофайловое решение |
| **Research** | Факты кодовой базы, факты платформы, риски, неизвестные | Финальный план реализации |
| **Vision** | Архитектура фичи, границы, долгосрочные решения | Чеклист выполнения |
| **Plan** | Точные файлы, контракты, последовательность, крайние случаи | Повторённый бизнес-контекст |
| **Brief** | Текущий пакет выполнения (пакет, ограничения, приёмка) | Полное обоснование дизайна |
| **Review Summary** | Находки, отклонения, вердикт | Доказательства QA |
| **Security Review** | Находки безопасности, границы доверия, go/no-go | Общее стилевое ревью |
| **QA** | Доказательства сценариев, покрытие, pass/fail | Переработка архитектуры |
| **Tasklist** | Кросс-фазный прогресс, готовность к релизу | Проза плана |

## Контракт метаданных

Каждый артефакт должен включать следующие заголовки:

```
Status: <gate status>
Ticket: <TICKET-ID>
Phase: <N> | feature
Lane: Professional | Critical
Workflow Version: 3
Owner: <role name>
```

Необязательные:
- `Date: YYYY-MM-DD`
- `Depends On: <artifact>`
- `Blocked Until: <gate>`

## Как писать каждый артефакт

### Idea
Пишите кратко и продуктово: что болит, зачем строить, что в скоупе, что вне.
Тест: *Если Implementer прочитает только idea, поймёт ли он, зачем фича существует?*

### PRD
Пишите верифицируемо: deliverables, позитивные сценарии, негативные сценарии, метрики успеха.
Тест: *Может ли QA верифицировать фазу, используя только этот документ?*

### Research
Пишите только факты: существующие файлы, текущие интерфейсы, ограничения платформы, факты зависимостей, риски.
Тест: *Это действительно найдено в коде/среде, или я начал проектировать?*

### Vision
Пишите долгосрочную архитектуру: границы, основные решения, давление на дизайн, ссылки на ADR.

### Plan
Пишите полностью решённо: точные файлы, точные контракты, последовательность, проверки, известные крайние случаи.
Тест: *Может ли Implementer завершить фазу без новых архитектурных решений?*

### Brief
Пишите как пакет выполнения: текущий пакет, ограничения, приёмка, условия остановки.
Должен включать заголовки `Lane:` и `Goal:` — хук PostCompact извлекает их для восстановления контекста.

### Review Summary
Сначала пишите находки: блокирующие проблемы, важные находки, отклонения от плана, вердикт.

### QA
Сначала пишите доказательства: позитивное покрытие, негативное покрытие, ручные проверки, верификация реализации, вердикт.

### Security Review
Пишите только релевантное для безопасности: риск утечки, границы секретов/ключей, понижение авторизации, безопасность хранилища/сети.

## Структура рабочего пространства фичи

```
docs/<TICKET>/
├── .active_ticket
├── idea-<TICKET>.md
├── vision-<TICKET>.md
├── tasklist-<TICKET>.md
├── metrics.log
├── <TICKET>-phase-N-summary.md
├── phase/<TICKET>/phase-N.md
├── plan/<TICKET>-phase-N.md
├── prd/<TICKET>-phase-N.prd.md
├── research/<TICKET>-phase-N.md
├── qa/<TICKET>-phase-N.md
└── security/<TICKET>-phase-N.md   (только Critical)
```

Полный аннотированный пример см. [[../Examples/Feature Workspace Structure]].
