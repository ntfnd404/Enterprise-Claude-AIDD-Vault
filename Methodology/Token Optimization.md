# Оптимизация токенов

## Философия

Каждый слой методологии спроектирован так, чтобы минимизировать расход токенов без потери качества. Цель — держать контекстное окно Claude сфокусированным на том, что важно для текущей задачи.

## Оптимизация хуков

### async vs sync

| Режим | Когда использовать | Влияние на токены |
|---|---|---|
| `async: true` | Уведомительные хуки (уведомления жизненного цикла) | Неблокирующие, 0 токенов рассуждения |
| sync (по умолчанию) | Проверки гейтов, инъекция контекста, форматирование | Блокирует до завершения, может инъектировать токены |

6 из 12 событий хуков выполняются async. Только проверки гейтов и инъекция контекста требуют sync.

### statusMessage vs systemMessage

| Режим | Когда использовать | Влияние на токены |
|---|---|---|
| `statusMessage` | Хук информирует только пользователя (форматирование, спиннер валидации) | Показывает спиннер в UI, 0 токенов в контексте |
| stdout (systemMessage) | Claude нужна информация (восстановление контекста PostCompact) | Инъектируется в контекст, стоит токенов |

Правило: если хук информирует только пользователя, используйте `statusMessage`. Если Claude нужна информация, пишите в stdout.

### Компактные сообщения

Все systemMessages должны быть в одну строку, менее 20 токенов. Пример:

```
Context restored. Ticket: <TICKET> | Phase: N | Lane: Critical | Goal: <goal>.
```

### Экономия за сессию

| Оптимизация | Экономия |
|---|---|
| async для уведомительных хуков | ~150 tok |
| statusMessage вместо systemMessage | ~500 tok |
| Компактные однострочные systemMessages | ~300 tok |
| **Итого экономия на хуках** | **~950 tok** |

## Оптимизация скиллов

| Поле frontmatter | Что делает | Когда использовать |
|---|---|---|
| `effort: low` | Снижает глубину рассуждения | Механические скиллы (validate, run-checks) |
| `effort: medium` | Сбалансированное рассуждение | Скиллы загрузки контекста (start-phase) |
| `context: fork` | Выполняется в изолированном контексте | Скиллы, вывод которых не нужен в основном контексте |
| `paths` | Активация по файлам | Доменные скиллы, применимые только к определённым директориям |
| `disable-model-invocation` | Только ручной вызов | Все скиллы рабочего процесса (предотвращение случайного срабатывания) |

### Экономия за сессию

| Оптимизация | Экономия |
|---|---|
| `effort: low` на механических скиллах | ~800 tok |
| `context: fork` на validate/checks | ~600 tok |
| `paths` на доменных скиллах | ~400 tok на неиспользуемый скилл |
| **Итого экономия на скиллах** | **~1800 tok** |

## Оптимизация MCP

### Форматирование только изменённых файлов

Вместо форматирования всей кодовой базы форматируйте только файлы, изменённые в текущей сессии:

```bash
git diff --name-only --diff-filter=d HEAD | grep '\.dart$'
```

Передайте список файлов в параметр `paths` MCP или в CLI-форматтер.

### MCP-First с Bash-фоллбэком

Используйте MCP-инструменты для детерминированных операций (format, analyze, test). Откатывайтесь к bash-командам только когда MCP недоступен. MCP-вызовы дешевле, потому что не проходят через рассуждения Claude.

## Оптимизация артефактов

### Phase Brief как пакет выполнения

Phase brief — это минимально возможный документ, содержащий всё необходимое для Implementer. Он не повторяет plan или PRD — он ссылается на них. Это сводит к минимуму нагрузку на контекст Implementer.

### Восстановление через PostCompact

Когда происходит компактификация контекста, Claude теряет всю рабочую память. Хук PostCompact инъектирует одну строку с тикетом, фазой, полосой и целью. Затем Implementer перечитывает только brief и plan — не всю цепочку артефактов.

### Один документ — одна ответственность

Ни один артефакт не дублирует область другого. Это значит, что Claude никогда не загружает избыточную информацию. Каждая роль читает только нужные ей артефакты:

| Роль | Читает |
|---|---|
| Analyst | idea |
| Researcher | idea, PRD, кодовую базу |
| Planner | vision, PRD, research |
| Implementer | brief, plan, PRD, conventions, style |
| Reviewer | diff, plan, PRD, conventions |
| Security Reviewer | diff, plan, PRD, review summary |
| QA | PRD, phase, plan, review/security |

## Чистая экономия

| Слой | Экономия/сессия |
|---|---|
| Хуки | ~950 tok |
| Скиллы | ~1800 tok |
| MCP + артефакты | ~450 tok |
| **Итого** | **~3200 tok** |

При контекстном окне в 200k это ~1.6% экономии. Реальный эффект в том, что контекст остаётся сфокусированным на текущей работе, а не на накладных расходах процесса.
