# Feature Workspace Structure

## Обзор

Каждая фича получает собственное рабочее пространство в `docs/<TICKET>/`. Эта директория является локальной для ветки и очищается перед мержем в основную ветку. Долгоживущие решения промоутятся в `docs/project/` до мержа.

## Полное дерево файлов

```
docs/<TICKET>/
├── .active_ticket
├── idea/
│   └── <TICKET>.idea.md
├── vision/
│   └── <TICKET>.vision.md
├── tasklist/
│   └── tasklist-<TICKET>.md
├── prd/
│   └── <TICKET>-phase-N.prd.md
├── research/
│   └── <TICKET>-phase-N.research.md
├── plan/
│   └── <TICKET>-phase-N.md
├── brief/
│   └── <TICKET>-phase-N.brief.md
├── phase/
│   └── <TICKET>/
│       └── phase-N.md
├── summary/
│   └── <TICKET>-phase-N.summary.md
├── qa/
│   └── <TICKET>-phase-N.qa.md
├── security/
│   └── <TICKET>-phase-N.security.md
└── adr/
    └── <TICKET>-NNN.adr.md
```

## Назначение каждой директории

| Директория | Назначение | Кто создает/владеет | Когда создается |
|---|---|---|---|
| `.active_ticket` | Содержит ID тикета. Хук PreToolUse использует его для проверки гейта -- блокирует редактирование, если файл отсутствует | Скилл `/aidd-new-ticket` | При создании тикета |
| `idea/` | Зачем фича существует: проблема, скоуп, цели бизнеса, критерии приёмки, выбор полосы (Lane) | Пользователь + analyst | При создании тикета |
| `vision/` | Архитектура фичи: границы, ключевые решения, давление на дизайн, ссылки на ADR | researcher | После исследования фазы |
| `tasklist/` | Кросс-фазный прогресс: какие фазы запланированы, какие завершены, готовность к релизу | planner, обновляет implementer | При планировании первой фазы |
| `prd/` | Требования фазы: deliverables, позитивные/негативные сценарии, метрики успеха, ограничения | analyst | В начале каждой фазы |
| `research/` | Факты кодовой базы: существующие файлы, интерфейсы, ограничения платформы, риски, неизвестные | researcher | После PRD, перед планом |
| `plan/` | Точный дизайн реализации: файлы, контракты, последовательность, крайние случаи | planner | После исследования |
| `brief/` | Пакет выполнения: текущий батч, ограничения, приёмка, условия остановки. Должен содержать заголовки `Lane:` и `Goal:` для хука PostCompact | planner | Вместе с планом |
| `phase/` | Исполнительный пакет фазы: содержит `Lane:` и `Goal:`, используется хуком PostCompact для восстановления контекста при сжатии | planner, обновляет implementer | При старте фазы |
| `summary/` | Результаты ревью: находки, отклонения от плана, блокирующие проблемы, вердикт | reviewer | После завершения реализации |
| `qa/` | Доказательства QA: покрытие сценариев, ручные проверки, верификация реализации, вердикт | qa | После ревью |
| `security/` | Находки безопасности: риск утечки, границы секретов, понижение авторизации, go/no-go. Создается **только для полосы Critical** | security-reviewer | После ревью, перед QA |
| `adr/` | Записи архитектурных решений. В Enterprise -- обязательны для архитектурных решений. В Lite/Standard -- опционально | planner или implementer | При принятии архитектурного решения |

## Что пропускается в уровне Lite

В [[../Methodology/Tier System|Lite]] рабочее пространство значительно упрощается:

| Артефакт | Lite | Standard / Enterprise |
|---|---|---|
| `.active_ticket` | Есть | Есть |
| `idea/` | Есть | Есть |
| `vision/` | Нет (нет агента researcher) | Есть |
| `tasklist/` | Есть | Есть |
| `prd/` | Нет (включен в plan) | Есть |
| `research/` | Нет (включен в plan) | Есть |
| `plan/` | Есть (объединяет research + plan) | Есть |
| `brief/` | Есть | Есть |
| `phase/` | Есть | Есть |
| `summary/` | Нет | Есть |
| `qa/` | Есть | Есть |
| `security/` | Нет (нет агента security-reviewer) | Только Critical |
| `adr/` | Опционально | Standard: опционально, Enterprise: обязательно |

Итого: ~4 артефакта на фазу в Lite против ~7+ в Standard/Enterprise.

## Ключевые правила

- Директория `docs/<TICKET>/` **никогда не мержится** в основную ветку
- Каждый артефакт владеет ровно одним типом знания -- см. [[../Methodology/Artifact Contracts]]
- Каждый артефакт должен содержать метаданные: `Status`, `Ticket`, `Phase`, `Lane`, `Workflow Version`, `Owner`
- Долгосрочные решения промоутятся в `docs/project/` перед мержем ветки
