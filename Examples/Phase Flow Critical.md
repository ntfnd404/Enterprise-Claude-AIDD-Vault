# Phase Flow Critical

## Сценарий

Фича: `PROJ-0003` -- добавление шифрования локального хранилища приватных ключей.
Полоса: **Critical** (криптография + безопасное хранение + ключевой материал).
Фаза: **1**.

## Предусловия

- Идея заполнена, `Lane: Critical` установлен в статусе `IDEA_READY`
- Ветка создана: `git checkout -b PROJ-0003-encrypted-key-storage`
- Рабочее пространство создано: `/aidd-new-ticket PROJ-0003`

## Отличия от Professional

Прежде чем показать полный поток, вот ключевые различия:

| Аспект | Professional | Critical |
|---|---|---|
| Размер батчей | 2-5 связанных задач | Меньше, чем в Professional |
| Security review | Нет | Обязательный шаг между review и QA |
| Артефакт `security/` | Не создается | Обязателен |
| Предположения безопасности | Не требуются в плане | Должны быть явно указаны в плане |
| Пробелы проектирования | Можно закрывать по ходу | Должны быть закрыты до реализации |
| Блокировка QA | Нет | QA заблокирован, если security review вернул блокировку |

## Полный поток фазы

### Шаг 1: Старт фазы

Пользователь вводит:

```text
/aidd-start-phase 1
```

### Шаг 2: Analyst создает PRD

Агент analyst создает `docs/PROJ-0003/prd/PROJ-0003-phase-1.prd.md`:

- **Deliverables**: зашифрованное хранилище ключей, интерфейс доступа, миграция существующих данных
- **Позитивный сценарий**: приложение сохраняет ключ -- данные зашифрованы на диске -- при чтении расшифровываются корректно
- **Негативный сценарий**: неверный пароль -- расшифровка отклонена с понятной ошибкой
- **Ограничение безопасности**: ключи не должны появляться в логах, памяти дольше необходимого, в stack traces
- **Метрика успеха**: все ключи зашифрованы AES-256-GCM, миграция прозрачна для пользователя

Статус: `PRD_APPROVED`.

### Шаг 3: Researcher собирает контекст

Агент researcher создает `docs/PROJ-0003/research/PROJ-0003-phase-1.research.md`:

- Текущее хранилище: `flutter_secure_storage` используется, но ключи хранятся как plain text внутри
- Платформенные ограничения: Android Keystore доступен через `flutter_secure_storage`, iOS Keychain аналогично
- Зависимости: пакет `encrypt` v5.0 уже в `pubspec.yaml`
- Риск: миграция существующих plain-text ключей должна быть атомарной -- частичная миграция = потеря данных
- Неизвестное: поведение при смене пароля устройства на Android

Обновляет `docs/PROJ-0003/vision/PROJ-0003.vision.md`.

### Шаг 4: Planner создает план

Агент planner создает `docs/PROJ-0003/plan/PROJ-0003-phase-1.md` с **явно указанными предположениями безопасности**:

**Предположения безопасности (обязательно для Critical):**
- Ключевой материал обнуляется в памяти сразу после использования
- Производные ключи шифрования генерируются через PBKDF2 с минимум 600 000 итерациями
- Неудачная дешифровка не раскрывает причину ошибки (timing attack mitigation)
- Миграция выполняется в одной транзакции -- rollback при любой ошибке

**План реализации:**
1. Создать `EncryptedKeyStorage` интерфейс в `packages/domain/`
2. Реализовать через `flutter_secure_storage` + AES-256-GCM в `packages/data/`
3. Создать `KeyMigrationService` для миграции plain-text -> encrypted
4. Обновить `KeyRepository` для использования нового хранилища
5. Написать тесты для каждого граничного случая

Также создает бриф и обновляет tasklist.

Статус: `PLAN_APPROVED` + `TASKLIST_READY`.

### Шаг 5: Implementer работает маленькими батчами

В Critical батчи меньше -- каждый фокусируется на одном аспекте безопасности.

#### Батч 1: Интерфейс зашифрованного хранилища

Implementer предлагает:

> **Батч 1**: Создать интерфейс `EncryptedKeyStorage` в `packages/domain/` с методами `store()`, `retrieve()`, `delete()`, `exists()`. Обновить экспорты.

Пользователь одобряет. Реализация + проверки:

```text
/aidd-run-checks
```

#### Батч 2: Реализация шифрования

Implementer предлагает:

> **Батч 2**: Реализовать `EncryptedKeyStorageImpl` с AES-256-GCM через `flutter_secure_storage`. Обнуление ключевого материала после использования.

Пользователь одобряет. Реализация + проверки:

```text
/aidd-run-checks
```

#### Батч 3: Сервис миграции

Implementer предлагает:

> **Батч 3**: Создать `KeyMigrationService` с атомарной миграцией plain-text -> encrypted. Rollback при ошибке. Unit-тесты для успешной миграции, частичного сбоя, повторного запуска.

Пользователь одобряет. Реализация + проверки:

```text
/aidd-run-checks
```

#### Батч 4: Интеграция и обновление репозитория

Implementer предлагает:

> **Батч 4**: Обновить `KeyRepository` для использования `EncryptedKeyStorage`. Обновить DI-конфигурацию. Обновить экспорты.

Пользователь одобряет. Реализация + проверки:

```text
/aidd-run-checks
```

### Шаг 6: Reviewer проверяет

Агент reviewer анализирует все изменения:

- Реализация соответствует плану
- Конвенции соблюдены
- Нет регрессий
- **Подготовка к security review**: reviewer отмечает области, требующие внимания security-reviewer
- Вердикт: `REVIEW_OK`

Результат: `docs/PROJ-0003/summary/PROJ-0003-phase-1.summary.md`.

### Шаг 7: Security Reviewer проверяет (только Critical)

Агент security-reviewer проводит проверку безопасности:

- Ключевой материал обнуляется после использования -- PASS
- Нет логирования чувствительных данных ни в одном слое -- PASS
- Производные ключи используют PBKDF2 с достаточными итерациями -- PASS
- Миграция атомарна, rollback работает -- PASS
- Нет небезопасных запасных путей (fallback) -- PASS
- Сообщения об ошибках не раскрывают внутреннее состояние -- PASS
- Приватный материал не утекает вверх через границы слоёв -- PASS
- Вердикт: `SECURITY_REVIEW_OK`

Результат: `docs/PROJ-0003/security/PROJ-0003-phase-1.security.md`.

**Блокирующее правило**: если security review вернул блокировку, переход к QA невозможен. Implementer должен исправить проблемы и пройти security review повторно.

### Шаг 8: QA проверяет сценарии

Агент qa запускает верификацию:

- **PS-1**: Сохранить ключ -- данные зашифрованы на диске -- PASS
- **PS-2**: Прочитать ключ -- расшифровка корректна -- PASS
- **PS-3**: Миграция plain-text ключей -- все ключи зашифрованы -- PASS
- **NE-1**: Неверный пароль -- дешифровка отклонена -- PASS
- **NE-2**: Частичный сбой миграции -- rollback, данные не потеряны -- PASS
- **IV-1**: Статический анализ чист -- PASS
- **IV-2**: Нет чувствительных данных в логах -- PASS
- **IV-3**: Все unit-тесты проходят -- PASS
- Вердикт: `QA_PASS`

Результат: `docs/PROJ-0003/qa/PROJ-0003-phase-1.qa.md`.

### Шаг 9: Завершение фазы

Пользователь вводит:

```text
/aidd-complete-phase 1
```

Скилл проверяет полную цепочку гейтов Critical: `REVIEW_OK` + `SECURITY_REVIEW_OK` + `QA_PASS`. Фаза закрывается.

## Полная цепочка гейтов Critical

```
IDEA_READY → PRD_APPROVED → RESEARCH_DONE → VISION_SET →
PLAN_APPROVED → TASKLIST_READY → IMPLEMENT_STEP_OK →
REVIEW_OK → SECURITY_REVIEW_OK → QA_PASS → PHASE_DONE
```

Обратите внимание на дополнительный гейт `SECURITY_REVIEW_OK` между `REVIEW_OK` и `QA_PASS`.
