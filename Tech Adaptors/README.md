# Технические адаптеры

## Что такое адаптер

Адаптер -- это набор наложений (overlays), специфичных для технологического стека. Адаптер дополняет универсальные шаблоны AIDD правилами, проверками и инструментами конкретного стека.

Адаптер НЕ заменяет универсальный воркфлоу -- он расширяет его деталями, которые имеют смысл только в контексте конкретной технологии.

## Структура адаптера

Каждый адаптер -- это папка с 5 файлами:

```
Tech Adaptors/<Stack-Name>/
├── conventions-overlay.md    # Архитектурные правила стека
├── code-style-overlay.md     # Стиль кода стека
├── run-checks-overlay.md     # Конвейер проверок (format→analyze→test)
├── post-edit-format.sh       # Хук PostToolUse для автоформатирования
└── mcp-config.md             # Конфигурация MCP-сервера стека
```

| Файл | Назначение |
|---|---|
| `conventions-overlay.md` | Архитектурные правила: слои, зависимости, паттерны, жёсткие запреты |
| `code-style-overlay.md` | Стиль кода: форматирование, именование, порядок членов класса |
| `run-checks-overlay.md` | Конвейер проверок: `format` → `analyze` → `lint` → `test` |
| `post-edit-format.sh` | Скрипт автоформатирования, вызываемый после каждого редактирования файла |
| `mcp-config.md` | Инструкция по настройке MCP-сервера для стека (если есть) |

## Как адаптер применяется

При `/aidd-init --adaptor <name>`:

1. `conventions-overlay.md` → добавляется в `docs/project/conventions.md`
2. `code-style-overlay.md` → добавляется в `docs/project/code-style-guide.md`
3. `run-checks-overlay.md` → используется для генерации `.claude/aidd-checks.sh`
4. `post-edit-format.sh` → копируется в `.claude/hooks/post-edit-format.sh` и регистрируется как `PostToolUse` хук в `settings.json`
5. `mcp-config.md` → генерирует `.mcp.json` в корне проекта

Результат: проект получает полностью настроенный рантайм AIDD с правилами, проверками и автоформатированием, специфичными для стека.

## Типы адаптеров

Адаптеры бывают двух видов:

| Тип | Назначение | Пример |
|---|---|---|
| **tech-stack** | Правила и инструменты конкретного технологического стека (язык + фреймворк) | `Flutter-Dart` |
| **domain** | Правила конкретной предметной области, не зависят от стека | `Crypto-Wallet` |

В одном проекте применяются **один tech-stack-адаптер** + **ноль или несколько domain-адаптеров**. Domain-адаптеры комбинируются со стеком: tech-stack отвечает за архитектуру, форматирование, MCP и проверки; domain добавляет правила и acceptance criteria для своей области.

| Адаптер | Тип | Назначение | Статус |
|---|---|---|---|

## Как создать свой адаптер

1. Создайте папку `Tech Adaptors/<Your-Stack>/`
2. Создайте 5 файлов по структуре выше
3. Опишите `conventions-overlay.md` -- архитектурные правила: слои, граф зависимостей, жёсткие запреты
4. Опишите `code-style-overlay.md` -- стиль кода с примерами "правильно / неправильно"
5. Опишите `run-checks-overlay.md` -- конвейер `format` → `analyze` → `lint` → `test` с MCP-first подходом и bash-фоллбэком
6. Напишите `post-edit-format.sh` -- скрипт должен:
   - Принимать JSON на stdin (с полем `tool_input.file_path`)
   - Форматировать файл штатным форматтером стека
   - Возвращать exit code 0 при успехе
7. Опишите MCP-конфигурацию в `mcp-config.md` (если для стека существует MCP-сервер)
8. Зарегистрируйте адаптер в таблице выше

## Принципы

- Адаптер **НЕ дублирует** универсальный воркфлоу -- только дополняет его
- `conventions-overlay` и `code-style-overlay` -- это наложения (overlays), а не замена базовых файлов
- Проверки должны поддерживать MCP-first подход с bash-фоллбэком
- Автоформатирование должно работать через `PostToolUse` хук
- Каждый адаптер самодостаточен: все 5 файлов должны быть заполнены
- Domain-адаптер **никогда** не переопределяет MCP-конфигурацию или формат-хук стека -- эти файлы у domain-адаптера являются заглушками, описывающими, что владельцем остаётся tech-stack-адаптер
- При конфликте правил между стеком и доменом действует более строгое правило
